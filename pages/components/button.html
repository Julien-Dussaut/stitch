Accessibility

Accessibility is not an extra layer added after the fact, it is part of how the Button works. This section explains the decisions that make the component usable with a keyboard, clear across themes and states, and reliable for different interaction contexts.

Keyboard Navigation

The component relies on the native <button> element, which makes it keyboard accessible by default. It can be reached with Tab and activated with Enter or Space without requiring custom interaction logic

Screen Reader Support

The Button uses semantic HTML first, then adds ARIA only when a state needs extra context, such as loading. This keeps the component understandable for assistive technologies without duplicating native behavior.

Contrast and Visibility

Colors, focus states, and text visibility are driven by design tokens and verified against real theme values. This helps keep the Button readable and distinguishable across variants, backgrounds, and active themes.

Disabled State

Disabled buttons use the native disabled attribute so they are correctly exposed as unavailable and removed from keyboard interaction when needed.

Loading State

Loading buttons keep their structure readable and use additional accessible text when visual indicators alone would not provide enough context for assistive technologies.

Touch Targets

Spacing and sizing tokens help keep the Button comfortable to tap, making interactive targets easier to use on touch devices and in dense interfaces.
